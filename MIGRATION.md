# Migration Strategy: OrderProcessor to .NET 8

Given the strict "zero downtime" requirement and the tight coupling of legacy Razor pages and ASMX services to the `OrderProcessor`, we should replace the legacy system without a massive, risky cutover.

**Phase 1: Encapsulation (New API)**
We will build the modernized .NET 8 `OrderProcessingService` and expose it as a standalone REST endpoint (e.g., via a Minimal API). It will initially connect to the existing legacy database to prevent data synchronization issues.

**Phase 2: The Proxy (Legacy Update)**
In the legacy .NET 4.7 application, we will strip out the 1,800 lines of spaghetti code inside the existing `OrderProcessor`. In its place, we will implement a lightweight HTTP client that acts as a proxy, forwarding the `ProcessOrder` arguments as a JSON payload to our new .NET 8 API. 

**Phase 3: Shadowing & Cutover**
We will deploy the .NET 8 API alongside the legacy app. Using a feature flag, we will route a small percentage of production traffic to the new API while the rest remains on the old logic. Once validated, we dial up the API traffic to 100%. The legacy UI remains untouched, completely unaware that the backend has been modernized.

## Why This Fits
This approach completely satisfies the board's constraints. There is **zero downtime** because the legacy application never goes offline—it simply changes where it sends the data. Furthermore, by leaving the UI and ASMX services alone for now, we strictly limit the blast radius to just the core business logic, ensuring the 6-month deadline is highly achievable. 

## Leadership Risk: Distributed Network Failures
**Risk:** By splitting the application into a frontend web server (.NET 4.7) and a backend API (.NET 8), we are introducing network boundaries that did not exist when everything ran in a single memory space. 

**Impact:** If the new .NET 8 API successfully processes a payment and saves the order, but the HTTP response back to the legacy proxy drops due to a transient network timeout, the legacy app might show a generic crash screen to the parent. The parent may hit "Refresh" and retry, leading to double-billing and massive customer service headaches during the August peak.

**Mitigation:** We must implement strict **idempotency keys** (using a unique ID generated by the frontend for each checkout attempt) so the payment gateway and database reject duplicate processing attempts during network retries.
